<p>Bagi kamu yang merupakan Web Developer ataupun kamu yang pernah belajar tentang Web Programming, pasti pernah menggunakan Cookie dan Session, misalnya pada saat membuat fitur Login, serta juga fitur seperti <em>Remember me</em> yang terdapat pada halaman Login. Mungkin diantara kalian ada yang berfikir bahwa Cookie dan Session adalah sesuatu yang berbeda, Session ya Session, Cookie ya Cookie (tidak memiliki hubungan apapun), saya dulu juga berfikir bahwa Session itu langsung tahu aja bahwa pengguna itu sudah Login atau belum. Tapi ternyata, Session itu tidak memiliki sesuatu hal yang <em>magic</em>, sebenarnya dibelakang layar Session menggunakan Cookie juga.</p>
<p>Menurut saya memahami cara kerja Cookie dan Session merupakan hal yang penting untuk seorang Web Developer, dulu waktu awal-awal belajar Web Programming, karena tidak begitu memahami Cookie dan Session, ketika saya yang biasanya menggunakan PHP dengan fitur Session-nya untuk membuat sistem Login, lalu mencoba membuat sistem Login dengan JavaScript dan NodeJs, menjadi bingung, karena di NodeJS tidak ada fitur Session bawaan seperti di PHP, di NodeJS kita biasanya menggunakan library pihak ketiga dari npm. Lebih bingung-nya lagi Session di NodeJS itu kok menggunakan Cookie? dan akhirnya saya berkesimpulan bahwa ini bukan Session, karena tidak sama seperti Session di PHP yang biasanya saya gunakan. Padahal jika memahami cara kerja Cookie dan Session sebelumnya, hal itu adalah benar fitur Session yang biasanya saya gunakan di PHP.</p>

<h2>Apa itu Cookie dan Bagaimana Cara Kerjanya?</h2>
<p>Cookie adalah kumpulan data kecil yang berisi informasi yang dikirim oleh Server ke Client (Browser). Cookie juga dikenal sebagai <em>Web Cookie</em>, <em>Browser Cookie</em>, atau <em>Internet Cookie</em>.</p>
<p>Cara kerja Cookie secara umum yaitu, Browser mengirim HTTP Request ke Server, lalu Server akan menyertakan header <code>Set-Cookie</code> (yang berisi data cookie-nya) pada HTTP Response, kemudian Browser akan menyimpan data yang terdapat pada header <code>Set-Cookie</code>, lalu pada request-request berikutnya, Browser akan menyertakan data Cookie tersebut pada HTTP Request header <code>Cookie</code>. Berikut jika digambarkan dalam bentuk diagram:
<img src="/posts/bagaimana-cara-kerja-cookie-dan-session/cookie-diagram.svg" alt="Cookie Diagram" width="802" height="441" loading="lazy" decoding="async"><!--rehype:width=802&height=441&loading=lazy&decoding=async--></p>
<p>Berikut adalah contoh dimana kita misalnya membuat Cookie "session identifier" dengan nama <code>SIDR</code> dan value <code>31d4d96e407aad42</code>:</p>
<pre><code class="language-http">Set-Cookie: SIDR=31d4d96e407aad42
</code></pre>
<p>Lalu pada request-request berikutnya Browser akan menyertakan header <code>Cookie</code> seperti ini:</p>
<pre><code class="language-http">Cookie: SIDR=31d4d96e407aad42
</code></pre>
<p>Dalam membuat Cookie kamu harus memperhatikan batas (<em>limit</em>) yang telah ditentukan, yaitu setidaknya maksimal ukuran per Cookie adalah 4096 bytes, ukuran tersebut tidak hanya dihitung dari value Cookie, tetapi juga nama dan atribut-atributnya. Selain itu juga, setidaknya per domain maksimal hanya 50 Cookie. Batasan tersebut adalah sebagaimana ditetapkan pada <a href="https://www.rfc-editor.org/rfc/rfc6265.html#page-27">RFC 6265</a>. Batasan tersebut juga bisa berbeda disetiap browser, tetapi biasanya perbedaannya tidak terlalu jauh.</p>
<p>Dalam membuat Cookie juga tidak disarankan memasukkan data terlalu banyak, karena semua data di Cookie akan selalu dikirim di setiap request, semakin banyak data, maka website kamu akan semakin lambat.</p>
<p>Secara default Cookie memiliki waktu <em>Expired</em> (kadaluarsa), yaitu "session", yang mana ini berarti bahwa Coookie akan dihapus ketika sesi (<em>session</em>) berakhir, yaitu biasanya ketika Browser ditutup (<em>Browser closed</em>). Jika kamu ingin misalnya data Cookie tetap ada walaupun Browser ditutup, maka bisa mengatur waktu kadaluarsa Cookie pada atribut <code>Expires</code>, contoh:</p>
<pre><code class="language-http">Set-Cookie: SIDR=31d4d96e407aad42; Expires=Wed, 09 Jan 2024 10:18:14 GMT
</code></pre>
<blockquote>
<p><strong>Catatan:</strong> Titik koma (;) digunakan sebagai pemisah antara atribut, maka dari itu nama Cookie tidak bisa mengandung karakter tersebut, lebih detailnya bisa kunjungi <a href="https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Set-Cookie#cookie-namecookie-value">cookie-name</a>.</p>
</blockquote>
<p>Waktu kadaluarsa ini harus dalam timezone GMT (<em>Greenwich Mean Time</em>), karena HTTP date selalu dibuat atau direpresentasikan dalam GMT, tidak pernah dalam waktu local. Dan format waktu kadaluarsa ini berdasarkan <a href="https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Date">spesifikasi HTTP date</a> adalah:</p>
<pre><code class="language-http">&#x3C;day-name>, &#x3C;day> &#x3C;month> &#x3C;year> &#x3C;hour>:&#x3C;minute>:&#x3C;second> GMT
</code></pre>
<p>Sebenarnya, masalah waktu kadaluarsa ini tidak perlu terlalu dipusingkan, kamu hanya perlu mengikuti format diatas dan Browser akan otomatis melakukan validasi apakah Cookie tersebut sudah kadaluarsa atau belum, relatif terhadap waktu client (Browser) dimana Cookie disimpan. Dengan kata lain Browser akan secara otomatis menyesuaikan timezone waktu kadaluarsa Cookie dengan timezone Browser, yang biasanya sama dengan pengaturan timezone di komputer kamu. Untuk lebih mudah dalam menentukan waktu kadaluarsa, kamu bisa menentukan waktu kadaluarsa dalam timezone saat ini, misalnya <code>Asia/Jakarta</code>, lalu konversi ke dalam timezone GMT, biasanya untuk setiap bahasa pemrograman, sudah menyediakan fungsi bawaan untuk kebutuhan seperti konversi timezone. Bahkan di bahasa pemrograman seperti PHP misalnya, ketika membuat Cookie, kamu tidak perlu memikirkan masalah timezone, karena kamu hanya perlu memasukkan waktu kadaluarsa dalam format Epoch Time, misalnya kamu ingin Cookie kadaluarsa dalam waktu 30 hari, maka kamu bisa membuat Epoch Time untuk 30 hari dari hari sekarang dengan kode <code>time()+60*60*24*30</code>.</p>
<p>Sebagai catatan, Browser mungkin menghapus Cookie sebelum waktu kadaluarsanya jika Cookie yang disimpan telah melebihi batas (<em>limit</em>), atau jika pengguna secara manual menghapus Cookie-nya, misalnya pengguna menghapus Cookie melalui Browser secara langsung atau ketika pengguna menggunakan software untuk membersihkan Sistem dan ruang penyimpanan, semacam BleachBit atau CCleaner.</p>
<p>Selain waktu kadaluarsa, kamu juga bisa mengatur <em>Scope</em> (Cangkupan) dari Cookie dengan menggunakan atribut <code>Path</code> dan <code>Domain</code>, sehingga nantinya Cookie hanya akan disertakan pada HTTP Request, ketika target dari HTTP Request sesuai dengan Path (serta sub-pathnya) dan Domain (serta sub-domainnya) yang kamu atur. Sebagai contoh misalnya kamu ingin suatu Cookie hanya disertakan pada HTTP Request ketika target-nya menuju ke Path <code>/cookie</code> (serta sub-pathnya) dari Domain <code>coba.rezafikkri</code> (serta sub-domainnya), maka atur seperti ini:</p>
<pre><code class="language-http">Set-Cookie: SIDR=31d4d96e407aad42; Path=/cookie; Domain=coba.rezafikkri
</code></pre>
<p>Maka nanti ketika kamu mengakses halaman <code>coba.rezafikkri/cookie</code>, <code>coba.rezafikkri/cookie/other</code>, atau <code>cookie.coba.rezafikkri/cookie</code> dan sebagainya, Cookie akan terus disertakan pada setiap HTTP Request, tetapi ketika kamu mengakses halaman <code>coba.rezafikkri</code>, <code>coba.rezafikkri/other</code> atau <code>cookie.coba.rezafikkri</code> misalnya, Cookie tidak akan disertakan pada HTTP Request, karena walaupun Domain dari halaman-halaman tersebut sesuai dengan yang kita atur, tetapi pada bagian Path-nya tidak sesuai, karena Path-nya bukan <code>/cookie</code> dan bukan sub-path dari <code>/cookie</code>.</p>
<p>Atau contoh lainnya, jika kamu ingin Cookie disertakan untuk setiap Request ke semua Path dari Domain <code>coba.rezafikkri</code> (serta sub-domainnya), maka kamu bisa atur seperti ini:</p>
<pre><code class="language-http">Set-Cookie: SIDR=31d4d96e407aad42; Path=/; Domain=coba.rezafikkri
</code></pre>
<p>Maka Cookie akan disertakan untuk setiap HTTP Request yang mengarah ke: <code>coba.rezafikkri</code>, <code>cookie.coba.rezafikkri</code>, <code>coba.rezafikkri/cookie</code>, <code>cookie.coba.rezafikkri/other</code> dan sebagainya.</p>
<p>Pada intinya untuk domain, jika kamu menentukan nilai dari atribut <code>Domain</code>, maka Cookie hanya akan disertakan untuk HTTP Request yang mengarah ke domain tersebut dan semua subdomain-nya. Tetapi jika kamu tidak menentukan nilai dari atribut <code>Domain</code>, maka Cookie hanya akan disertakan untuk HTTP Request yang mengarah ke domain dimana Cookie dibuat (karena nilai default dari atribut <code>Domain</code> adalah domain dimana Cookie dibuat) dan tidak untuk semua subdomain-nya. Dan mau kamu menentukan atau tidak nilai dari atribut <code>Domain</code>, Cookie tidak akan pernah disertakan pada HTTP Request yang mengarah ke parent domain dari domain tersebut (jika domain tersebut memiliki parent domain), misalnya kamu membuat Cookie di domain <code>cookie.coba.rezafikkri</code>, maka parent domainnya adalah <code>coba.rezafikkri</code>, sehingga jika HTTP Request mengarah ke parent domain tersebut, Cookie tidak akan disertakan.</p>
<p>Sedangkan untuk path, mau kamu menentukan atau tidak nilai dari atribut <code>Path</code>, maka Cookie hanya akan disertakan untuk HTTP Request yang mengarah ke path tersebut dan semua subpath-nya. Nilai default dari atribut <code>Path</code> adalah path dimana Cookie dibuat.</p>
<p>Dan tentunya Cookie hanya akan disertakan jika path dan domain dari HTTP Request tersebut cocok dengan nilai dari atribut <code>Path</code> dan <code>Domain</code>.</p>
<p>Nilai untuk atribut <code>Domain</code> hanya bisa domain dimana kamu membuat Cookie, atau parent domain dari domain tersebut (jika domain tersebut memiliki parent domain), tidak bisa domain lain yang benar-benar berbeda. Contoh, jika kamu membuat Cookie pada domain <code>coba.rezafikkri</code>, maka nilai dari atribut <code>Domain</code> hanya bisa domain itu saja (karena domain tersebut merupakan <a href="https://www.siteground.com/kb/what-is-a-root-domain/">root domain</a> yang tidak memiliki parent domain) dan tidak bisa domain lain yang benar-benar berbeda seperti <code>php.net</code> misalnya, namun jika kamu membuat Cookie pada subdomain <code>cookie.coba.rezafikkri</code>, maka nilai dari atribut <code>Domain</code> hanya bisa subdomain itu saja dan parent domainnya, yaitu <code>coba.rezafikkri</code> dan tentu juga tidak bisa domain lain yang benar-benar berbeda.</p>
<p>Jika kamu ingin membuat lebih dari satu Cookie dalam satu HTTP Response, kamu bisa bisa mendefinisikan dua atau lebih header <code>Set-Cookie</code> sekaligus seperti ini:</p>
<pre><code class="language-http">Set-Cookie: SIDR=31d4d96e407aad42; Path=/; Domain=coba.rezafikkri
Set-Cookie: lang=en-US; Path=/; Expires=Wed, 09 Jan 2024 10:18:14 GMT
</code></pre>
<p>dan pada request-request berikutnya Browser akan menyertakan header <code>Cookie</code> seperti ini:</p>
<pre><code class="language-http">Cookie: SIDR=31d4d96e407aad42; lang=en-US
</code></pre>
<p>Selain mengatur waktu kadaluarsa dan scope dari Cookie, kamu juga bisa mengatur supaya Cookie tidak bisa diakses melalui JavaScript yang ada di Browser (seperti menggunakan <code>document.cookie</code>), yaitu dengan mengatur atribut <code>HttpOnly</code>, seperti ini:</p>
<pre><code class="language-http">Set-Cookie: SIDR=31d4d96e407aad42; Path=/; HttpOnly; Expires=Wed, 12 Des 2024 10:18:14 GMT
</code></pre>
<p>Jika kamu membuat fitur Login yang bertahan lama, sehingga walaupun Browser ditutup pengguna akan tetap login, yang sekarang banyak diterapkan oleh perusahaan besar seperti Google misalnya dan kamu menggunakan Cookie untuk menentukan pengguna sudah login atau belum, maka sangat disarankan untuk mengatur atribut <code>HttpOnly</code>. Dengan mengatur atribut <code>HttpOnly</code> pada fitur login misalnya, akan membantu mencegah serangan yang bernama <a href="https://developer.mozilla.org/en-US/docs/Web/Security/Types_of_attacks#cross-site_scripting_xss">Cross-Site Scripting (XSS)</a> yang bisa mencuri Cookie yang digunakan untuk menentukan suatu pengguna sudah login atau belum.</p>
<p>Selain atribut <code>HttpOnly</code>, kamu juga bisa mengatur supaya Cookie hanya akan dikirim untuk Request yang menggunakan protocol HTTPS, yaitu dengan mengatur atribut <code>Secure</code>, seperti ini:</p>
<pre><code class="language-http">Set-Cookie: SIDR=31d4d96e407aad42; Path=/; Secure; HttpOnly; Expires=Wed, 12 Des 2024 10:18:14 GMT
</code></pre>
<p>Dengan mengatur atribut <code>Secure</code>, akan membantu mencegah penyerang <a href="https://en.wikipedia.org/wiki/Man-in-the-middle_attack">Man-in-the-middle</a> mengakses Cookie dengan mudah. Dan juga website yang menggunakan protocol HTTP tidak bisa membuat Cookie dengan atribut <code>Secure</code>. Tetapi jika kamu membuat website di localhost (dengan menggunakan host localhost dan tidak menggunakan <a href="https://httpd.apache.org/docs/2.4/vhosts/">virtual host</a>), Cookie akan tetap bisa dibuat dan dikirim, walaupun menggunakan protocol HTTP.</p>
<p>Terakhir, untuk menghapus Cookie, caranya cukup mudah, kamu hanya perlu mengatur waktu kadaluarsa menjadi sebelum waktu saat ini, sehingga nantinya Browser akan secara otomatis menghapus Cookie tersebut, karena dianggap sudah kadaluarsa. Tetapi perlu diingat, value dari atribut <code>Path</code> dan <code>Domain</code> harus sama dengan Cookie yang ingin dihapus. Misalnya saat membuat Cookie:</p>
<pre><code class="language-http">Set-Cookie: SIDR=31d4d96e407aad42; Path=/cookie; Domain=coba.rezafikkri
</code></pre>
<p>Lalu karena saat menghapus Cookie waktu saat ini adalah <code>Sun, 24 Nov 2024 03:42:10 GMT</code>, maka atur waktu kadaluarsa menjadi waktu sebelum itu dan jangan lupa pastikan value dari atribut <code>Path</code> dan <code>Domain</code> sama:</p>
<pre><code class="language-http">Set-Cookie: SIDR=; Path=/cookie; Domain=coba.rezafikkri; Expires=Sun, 24 Nov 2024 02:42:10 GMT
</code></pre>
<h2>Apa itu Session dan Bagaimana Cara Kerjanya?</h2>
<p>Setelah membahas mengenai Cookie, sekarang kita akan membahas mengenai Session, yang sebenarnya masih berkaitan dengan Cookie.</p>
<p>Session adalah cara atau teknik untuk menyimpan informasi client di Server. Session juga sebenarnya menggunakan Cookie, namun bedanya nilai expire dari cookie Session biasanya adalah "session", yang berarti jika sesi telah berakhir maka Session akan otomatis dihapus dan tidak seperti Cookie yang mana informasi-nya disimpan di Client (Browser), kalau Session, informasi-nya disimpan di Server, baik itu di dalam file, di dalam database dan lain-lain. Sedangkan yang dikembalikan oleh Server melalui header <code>Set-Cookie</code> pada HTTP Response adalah biasanya ID dari Sessionnya. Selain itu juga, Session tidak ada di spesifikasi HTTP, berbeda dengan Cookie yang mana ada di spesifikasi HTTP.</p>
<p>Cara kerja Session secara umum adalah, Browser mengirim HTTP Request ke Server, lalu Server akan membuat Session dan menyimpan Session tersebut, misalnya di database, selanjutnya Server akan mengirim HTTP Response ke Browser disertai header <code>Set-Cookie</code> yang berisi ID dari Session tersebut, pada request selanjutnya Browser akan menyertakan Cookie yang berisi ID dari Session, lalu Server akan mengecek (misalnya ke database) apakah Session dengan ID tersebut ada, jika ada maka data Session-nya akan diload sehingga bisa diakses di sisi server <em>(<a href="https://www.enonic.com/blog/what-is-the-difference-between-server-side-and-client-side">server side</a></em>). Berikut jika digambarkan dalam bentuk diagram:
<img src="/posts/bagaimana-cara-kerja-cookie-dan-session/session-diagram.svg" alt="Session Diagram" width="861" height="373" loading="lazy" decoding="async"><!--rehype:width=861&height=373&loading=lazy&decoding=async--></p>
<p>Seperti yang dijelaskan sebelumnya pada bagian Cookie, pertimbangkan juga untuk mengatur atribut <code>HttpOnly</code> dan <code>Secure</code> pada cookie Session.</p>
